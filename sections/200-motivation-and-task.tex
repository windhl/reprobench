\section{Motivation and Task}
% 现有benchmark均在环境准备好后才开始评估，\sys直接从CVE ID开始
Existing cybersecurity benchmarks measure post-environment capabilities: CyberGym and SEC-bench provide source-level projects~\cite{cybergym,secbench}; CVE-Bench provides prepared web targets~\cite{cvebench}; ExploitBench evaluates after a target is available~\cite{exploitbench}; K-Repro starts from kernel patch~\cite{krepro}. None answer whether agents can create the reproduction environment themselves. \sys{} asks that question directly: the input is a CVE identifier only; the goal is auditable evidence of reproduction on the real affected firmware service. IoT firmware provides a compact stress test---the agent must retrieve public information, locate firmware that may have disappeared, distinguish real firmware from HTML errors, unpack vendor-specific formats, reason about non-x86 binaries, and use rehosting tools. Each stage leaves auditable artifacts.

\sys{} defines six dependent stages: P1 Information Gathering, P2 Firmware Acquisition, P3 Firmware Extraction, P4 Binary Identification, P5 Service Rehosting, and P6 Vulnerability Triggering. The dependency order is strict: P6 is valid only if the real service from P5 is reachable, and P5 is meaningful only after the correct firmware and binary have been identified. This structure is essential for rejecting simulation substitution.
