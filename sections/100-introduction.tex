\section{Introduction}
%% 现有benchmark假设目标环境已准备好，真实漏洞复现从CVE起步更困难
LLM agents now plan, call tools, write code, and operate in software repositories with increasing autonomy, producing many benchmarks for software engineering and cybersecurity~\cite{swebench,programbench,cybergym,cvebench,secbench,bountybench,exploitgym,exploitbench,krepro}. 
%
A common protocol is that the evaluated targets should already have a repository, patch, container, harness, or running service. 
%
However, known-vulnerability reproduction, thought as a routine capability, in the wild usually begins in a less structured state.
%
For example, a security analyst may start with only a CVE identifier and then discover the affected product, locate the correct artifact, recover a runnable environment, and validate vulnerability behavior against the real target, i.e., producing a Proof-of-Concept(PoC). 

%
In this work, we focus on the IoT firmware vulnerability reproduction which makes the gap especially sharp in nowadays.
%
Specifically, one firmware is a proprietary binary blob compiled for MIPS or ARM, packed in vendor-specific formats, dependent on missing NVRAM state, and reachable only after \textit{rehosting}~\cite{greenhouse} under emulation. 
%
As a result, firmware vulnerability reproduction can be defined as an end-to-end artifact reconstruction and validation task.

%% 引入ReproBench：从CVE ID出发评估完整artifact-to-execution pipeline
To address the problem, we introduce \sys, a benchmark that evaluates whether LLM agents can move from a bare CVE identifier toward auditable evidence of IoT firmware vulnerability reproduction. 
%
\sys{} moves beyond prepared environments to evaluate the full artifact-to-execution pipeline, stressing capabilities that existing benchmarks totally ignore, including open-world information gathering, artifact provenance checking, cross-architecture binary analysis, long-horizon tool orchestration, and grounded validation. 


% 伪造复现容易，需要非二元评分+真实目标门控, we should re-write it to make it more clear
%%Our benchmark exposes a methodological problem, i.e., \textit{plausible-looking reproductions are easy to fabricate}. 
As one of our most important findings, \textit{emulating vulnerability, not emulating the environment} is a common failure behavior for most of current frontier models.
%
When rehosting is difficult, agents produce substitutes such as Python mock servers or synthetic C programs that crash. 
%
\sys{} uses non-binary, evidence-grounded scoring across six stages with real-target gating for rehosting and triggering.

% 贡献
Our contributions are as follows:
\begin{itemize}
\item We formulate CVE-only vulnerability reproduction as a from-scratch, long-horizon LLM agent benchmark.
\item We construct a 30-CVE public-evidence dataset balanced across buffer overflow, command injection, and authentication bypass.
\item We propose an evidence-grounded scoring framework with real-target gating across six dependent stages.

\end{itemize}
